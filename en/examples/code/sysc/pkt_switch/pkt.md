# Packet -- Packet Data Structure

## Software Analogy

`pkt` is a **message object** containing payload and routing metadata. Similar to:

```python
@dataclass
class Packet:
    data: int          # 8-bit payload (the actual data carried)
    id: int            # 4-bit sender ID (who sent it)
    dest0: bool        # Whether to send to port 0
    dest1: bool        # Whether to send to port 1
    dest2: bool        # Whether to send to port 2
    dest3: bool        # Whether to send to port 3
```

Or in networking terms, this is a simplified **Ethernet frame**: `data` is the payload, `id` is the source address, and `dest0`..`dest3` is the destination address (represented as a bitmask, supporting multicast).

## Structure Definition

```cpp
struct pkt {
    sc_int<8> data;    // 8-bit signed integer, packet data content
    sc_int<4> id;      // 4-bit signed integer, sender ID
    bool dest0;        // Destination 0 flag
    bool dest1;        // Destination 1 flag
    bool dest2;        // Destination 2 flag
    bool dest3;        // Destination 3 flag
};
```

### Field Descriptions

| Field | Type | Bits | Range | Purpose |
|-------|------|------|-------|---------|
| `data` | `sc_int<8>` | 8 | -128 ~ 127 | Data carried by the packet (payload) |
| `id` | `sc_int<4>` | 4 | -8 ~ 7 | Sender identifier (only 0-3 actually used) |
| `dest0` | `bool` | 1 | true/false | Whether to send to receiver 0 |
| `dest1` | `bool` | 1 | true/false | Whether to send to receiver 1 |
| `dest2` | `bool` | 1 | true/false | Whether to send to receiver 2 |
| `dest3` | `bool` | 1 | true/false | Whether to send to receiver 3 |

The entire packet is 16 bits, corresponding to the format in the README:

```
I<-ad(4)->I<-id(4)->I<---data(8)--->I
+-----------------------------------+
| | | | | | | | | | | | | | | | | | |
+-----------------------------------+
```

### Destination Bitmask Examples

| dest3 | dest2 | dest1 | dest0 | Meaning |
|-------|-------|-------|-------|---------|
| 0 | 0 | 0 | 1 | Unicast: send to receiver 0 only |
| 0 | 1 | 1 | 0 | Multicast: send to receiver 1 and 2 |
| 1 | 1 | 1 | 1 | Broadcast: send to all receivers |

The dest value generated by the sender ranges from 1-15 (`rand()%15+1`), guaranteeing at least one destination.

## Required SystemC Support Functions

To use a custom struct in `sc_signal<pkt>`, SystemC requires the following functions:

### `operator==`

```cpp
inline bool operator == (const pkt& rhs) const {
    return (rhs.data == data && rhs.id == id && ...);
}
```

`sc_signal` needs to compare old and new values to decide whether to trigger an event. If the new value is the same as the old value, no event is triggered.

**Software Analogy**: Like React's `shouldComponentUpdate()` -- the framework needs to know whether the value has actually changed to decide whether to notify listeners.

### `operator<<`

```cpp
inline ostream& operator << (ostream& os, const pkt&) {
    os << "streaming of struct pkt not implemented";
    return os;
}
```

Used for debug output. This is just a placeholder and does not actually print packet contents.

### `sc_trace`

```cpp
void sc_trace(sc_trace_file* tf, const pkt& a, const std::string& name) {
    sc_trace(tf, a.data, name + ".data");
    sc_trace(tf, a.id, name + ".id");
    sc_trace(tf, a.dest0, name + ".dest0");
    // ...
}
```

Used for waveform tracing. Each field of the struct is written separately to the trace file so it can be observed in a waveform viewer.

**Software Analogy**: Like structured logging -- recording each field of an object separately in the log, rather than just logging a `toString()` result.

## Design Observations

This packet structure reflects the hardware design philosophy of "everything is bits":
- The destination uses 4 separate `bool` fields instead of a single `int`, because in hardware each bit can be operated on independently
- `sc_int<8>` is used instead of `int`, explicitly specifying the bit width
- The entire struct maps precisely to 16 physical wires
